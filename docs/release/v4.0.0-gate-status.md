## GATE K — Threat-model note: stored per-analyst data

**Status:** ✅ MET

**Threat-model note:**
The v4.0.0 release introduces the first stored per-analyst data surface (PR2: preferences; PR3: saved views and query history). The following controls are in place and verified in code:

1. **Identity scoping (no IDOR):** Every row in `db_analyst_prefs.py` is keyed to the analyst identity resolved server-side from the session token via `auth_analyst.py`. The analyst identifier is never taken from the request body or query string — a client cannot supply or override their own owner key.

2. **Parameterized queries (no SQL injection):** All `db_analyst_prefs.py` and `saved_views.py` SQLite operations use parameterized queries (`?` placeholders with a separate params tuple). No string interpolation into SQL is present.

3. **No HTML sink (no stored XSS):** Preference values (time range, density, landing page) and saved-view names/filter JSON are stored and returned as JSON primitives or opaque strings. The React components (`AnalystPrefsContext`, saved-view panels) render these as text content — no `dangerouslySetInnerHTML`, no `innerHTML` assignment, no server-side HTML templating of stored values.

4. **Scope of stored data:** Only analyst-chosen UI preferences and filter sets are persisted. No event data, no credentials, no PII beyond the analyst's own session identity are stored in the per-analyst SQLite store.

**Residual risk:** Low. The stored data is analyst-chosen configuration; the primary risk (IDOR) is mitigated by server-side identity enforcement. The secondary risks (injection, XSS) are mitigated structurally.
